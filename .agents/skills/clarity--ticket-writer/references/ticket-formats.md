# Ticket Output Formats

## Format A — Trivial Requests (Tier 1)

Use this format for typos, simple text changes, color tweaks, and other cosmetic fixes.

```
**[Task]: {Clear, concise title}**

- **Complexity Tier:** 1
- **Location:** [Specific URL / Page]
- **Target File(s):** [Specific file path from codebase knowledge]
- **Action:** Change "[Old State/Text]" to "[New State/Text]".
```

### Example

```
**[Task]: Fix "Enterprize" typo on pricing page**

- **Complexity Tier:** 1
- **Location:** /pricing
- **Target File(s):** src/pages/pricing_tier.tsx
- **Action:** Change "Enterprize" to "Enterprise" (line 42).
```

---

## Format B — Standard and Complex Requests (Tier 2 and 3)

Use this format for bug fixes, feature enhancements, and complex features.

```
**[Feature/Bug]: {Clear, concise title}**

- **Complexity Tier:** [2 or 3]

**Description / User Story:**
As a [User Type], I want to [Action], so that [Benefit/Value].
[Add clear summary of the feature or bug. If a bug, include Steps to Reproduce, Expected vs Actual Behavior.]

**Acceptance Criteria:**

- **Scenario 1: [Happy Path]**
  - **Given** [initial context]
  - **When** [action occurs]
  - **Then** [expected outcome]

- **Scenario 2: [Error State/Edge Case]**
  - **Given** [initial context]
  - **When** [action occurs]
  - **Then** [expected outcome]

**Design / UX Requirements (if applicable):**
- Loading state: [Describe]
- Empty state: [Describe]
- Error state: [Describe]

**Technical Context (Generated by AI for Engineering):**
- *Target Modules/Files:* [List specific modules/files from codebase]
- *Implementation Notes:* [Technical constraints, APIs, database tables, or relevant architectural considerations based on codebase knowledge]
- *Assumptions Made:* [List any assumptions made under the Anti-Pedantry Rule]
```

### Example

```
**[Feature]: Add phone number login option**

- **Complexity Tier:** 3

**Description / User Story:**
As a returning user, I want to log in with my phone number,
so that I can access my account without remembering my email.

**Acceptance Criteria:**

- **Scenario 1: Happy Path — Phone Login**
  - **Given** a registered user with a verified phone number
  - **When** they enter their phone number and the SMS verification code
  - **Then** they are authenticated and redirected to the dashboard

- **Scenario 2: Unregistered Phone Number**
  - **Given** a phone number not associated with any account
  - **When** the user attempts to log in with it
  - **Then** they see "No account found for this number. Would you like to sign up?"

- **Scenario 3: SMS Delivery Failure**
  - **Given** a registered user requesting an SMS code
  - **When** the SMS provider returns an error
  - **Then** the user sees "Unable to send verification code. Please try again or use email login."

**Design / UX Requirements:**
- Loading state: Spinner on the "Send Code" button; button disabled during request.
- Empty state: N/A
- Error state: Inline error message below the phone input field.

**Technical Context (Generated by AI for Engineering):**
- *Target Modules/Files:* src/auth/login.tsx, src/api/auth.ts, src/services/sms.ts
- *Implementation Notes:* Current auth flow is email-only (src/auth/login.tsx). User model requires email (src/models/user.ts) — adding phone login requires making email optional or adding a phone field. SMS provider integration needed (no existing SMS service).
- *Assumptions Made:* Phone numbers will use E.164 format with country code selector. Rate limiting on SMS sends (max 3 per 10 minutes) assumed based on standard practice.
```
